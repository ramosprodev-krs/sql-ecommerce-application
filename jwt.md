# 🔐 JWT Architecture & Security Flow
This document details the internal structure and technical implementation of JSON Web Tokens (JWT) within the application's security layer.

## 1. 🏗️ The Token Structure

Every token generated by this API follows the RFC 7519 standard, composed of three Base64URL encoded segments:  

- Header: Specifies the token type and the hashing algorithm (e.g., HS256).
- Payload (Claims): The core data of the token. In this project, it contains:
  •sub: The subject (username).
  •iat: Issued-at timestamp.
  •exp: Expiration timestamp.
  •roles: The user's specific authorities (Permissions).
- Signature: Created by hashing the header and payload with a secret key. This ensures data integrity—if a single bit in the payload changes, the signature becomes invalid (the Butterfly Effect of data security).

## 2. 🛡️ Spring Security Integration
The application uses a Stateless Authentication model. Each request is independent and verified through a security filter chain.

### 2.1. The Authentication Filter
Before reaching any Controller, the request passes through a JWT Authentication Filter:  

- Extraction: It identifies the Authorization header.
- Format: It strictly validates the Bearer <token> format.
- Verification: The server validates the signature using the internal secret key.
- Context: If valid, the user's identity and roles are injected into the SecurityContextHolder.

## 3. Role-Based Access Control (RBAC)
The JWT acts as a digital passport, carrying specific clearance levels that determine which resources the user can interact with.

### 3.1. Role	Access Level	Description:  
``` 
ROLE_USER -	(Basic)	Can browse products, manage their own profile, and place orders.
ROLE_MANAGER -	(Intermediate)	Can manage post, update, and delete products of its own.
ROLE_ADMIN -	(Full)	Full system access, including user management and sensitive configurations.
```

## ⏳ 4. Security Constraints & Implementation
The security of the application is anchored in a Stateless Architecture, where the server delegates the responsibility of session state to the token itself. This is primarily handled by the SecurityConfig and TokenService classes.

1. Statelessness
The server does not persist tokens in a database (e.g., PostgreSQL). Instead, it relies on Cryptographic Trust.

    In the code: The SecurityConfig class defines the session creation policy as SessionCreationPolicy.STATELESS.
    Impact: Every request must carry its own "proof of identity," which the server verifies on the fly using the JWT.require() method.

3. Expiration (TTL)
Tokens are designed with a Short-Lived Strategy to mitigate the risk of token theft.

    In the code: The TokenService class calculates the exp (expiration) claim using Instant.now().plusSeconds(expiryTime).
    Impact: Once the expiration timestamp is reached, the JWTVerificationException is triggered within the filter, and the server automatically rejects the request, requiring a new login.

4. Secret Key & Hashing
The integrity of the entire system depends on a Private Secret Key that should never be hardcoded.
    In the code: The Algorithm.HMAC256(secret) method in your service uses a value retrieved via @Value("${api.security.token.secret}").
   
### 4.1 🛡️ Core Security Classes
In summary, the classes	responsibilities can be divided in:

1. SecurityFilter	Intercepts requests and validates the Bearer token.
2. TokenService	Handles the generation and verification of JWTs.
3. SecurityConfig	Configures the filter chain and endpoint permissions (RBAC).
